## 公网转发工具：FRP

**为什么需要公网转发？**

一个团体、组织内部有诸多互联的设备，比如PC、其它终端设备（如打印机）、服务器。为了保护这些设备，现代的计算机网络发展出了一套有效防御外部恶意攻击的手段，采用诸如防火墙、网络地址转换 NAT（Network Address Translation）（单独划分出一些只在内部网络起作用的ip地址）等手段，划分出了“**内部**”和“**外部**”的概念。这样的“一刀切”虽然很好地阻隔了外部攻击，但也反过来阻止了一些内网中服务、请求向外网的直接传播。

无论是用家用路由器、校园网中的 PC 做 MC 服务端，还是用从云服务商租来的服务器做 MC 服务端，这些设备总是处于这种阻隔保护之内的。因此，如果想让服务器真正提供服务，就应打通内外网之间的链接。

**公网转发技术**就像是给处于内网的服务器搭了一座桥，让外网的玩家也能访问到你位于内网的服务器。**FRP**（Fast Reverse Proxy）是一个轻量级的内网穿透**工具**，能把内网服务映射到公网。

FRP 只需要配置一个 `frpc.toml` 文件，就能快速实现转发。它还支持多种可靠传输协议（如 TCP、HTTP），性能可靠。对于使用**小型服务器和PC**作为 MC 服务端的腐竹们来说，**FRP**确实是简单好用的穿透工具。

**举个例子：**
假如你的服务器内网 IP 是 `192.168.1.2`，端口是 `11451`（在MC中可见的，发出多人游戏服务的端口，即内端口），你希望外网玩家通过 `1145` 端口访问（对外暴露的、外部玩家用来访问的端口，即外端口）。通过 FRP 配置后，外网的请求会被**转发**到你的服务器上，也就是说，你的服务器就可以为外网的访问者提供服务了。

详细配置请参考：[FRP 配置指南](./docs/server_arch/env_cfg/frp/FRP-doc.md)。

## 图形化管理面板：MCSM

**为什么需要图形化管理面板？**

有了打通内外网的手段，接下来我们就可以尝试运行 MC 服务端实例，为外网用户提供服务了。接下来的问题是，我们需要在服务器上准备好运行MC服务端的一切前置资源。问题来了：我是 Linux 苦手，对命令行一窍不通，我该怎么无痛准备服务器资源呢？

我们知道，运行 MC 服务的实例都在远程服务器上，因此，无论是开服时想要下载 MC 服务器所需的资源，还是后续管理服务器时，都必须通过某些远程连接的手段来控制服务器。先前，这个远程管理的任务都是通过连接到主机后(你可能听说过，所谓的 SSH)，使用命令行或者简单的远程文件管理器来进行的。这些东西对使用者提出了一定技术水平要求，不是那么的“**用户友好**”。

MCSM就很好地解决了这个问题。它提供一个网页界面，可以同时管理多个服务器实例，也可为不同用户分配不同的管理权限，方便团体管理。

详细操作请参考：[MCSM 使用教程](./docs/server_arch/env_cfg/mcsm/mcsm.md)。

## PVE 网络转发

**为什么需要 PVE 网络转发？**

当PC或者小型服务器的性能不能满足需求时，最好的办法是换功能更强大的服务器。然而，这些服务器往往并不只是为你一个人提供服务。直接在这些服务器中部署MC服务端服务可能**对目前在运行的其它任务有意想不到的影响**，可能对他人造成很大的困扰。即使你的钞能力能够盘下一整个大型服务器供你一人使用，出于方便管理、方便备份、方便迁移、方便关闭、方便删除的需要，也建议使用一些隔离的手段来单独划分出一个“空间”运行MC服务端。**虚拟机**（Virtual Machine, VM）就是用来在宿主机（电脑或服务器）上独立运行一个**完整的操作系统**的技术。像是在宿主机内额外寄生了一个小机器一样，所以叫虚拟机。如果你学过 Python，那么应该听说过，在运行某大型项目前，最好单独为它创建一个虚拟环境。虚拟机就像这种虚拟环境，里面包装了一个小的、完整的系统，我们可以在这里面部署 MC 服务端所需要的一切资源。这些操作不会对虚拟机以外的系统造成任何影响，将MC服务端与其它程序分隔开，且虚拟机方便管理、备份、迁移、关闭、删除。这种**方便别人也方便自己**的**封装**的方法，在实际项目中经常用到，我们以后还会遇到。

**注意**：尽管称虚拟机“独立”于宿主机的其它部分，但实际上，在设备的物理层**并没有也不可能**单独实现虚拟机。因此，实际运行时，虚拟机仍然需要占用宿主机的硬件资源，如CPU、内存、存储等。在管理多个MC服务端实例时，这一点要万分注意。

PVE（Proxmox VE）是一个企业级的虚拟化平台，适合部署大型的 MC 服务器。然而，正如前文所述，虚拟机通常运行在内网环境中，外网玩家无法直接访问。这时候就需要用 PVE 的网络转发功能，把外网的请求转发到虚拟机。重要的是，PVE 的转发规则可以实时生效，且通过严格的规则配置，可以避免误操作，安全性强。

**举个例子：**
假如你在 PVE 上运行了一台虚拟机，IP 是 `192.168.1.3`，端口是 `11451`。通过配置 `iptables`，你可以把外网的 `1145` 端口请求转发到虚拟机上。

详细配置请参考：[PVE 网络转发教程](./docs/server_arch/env_cfg/pve/网络转发.md)。

## Docker镜像

**为什么需要 Docker 镜像？**

Docker 镜像（Docker image）也是一种**封装**的技术。网上的教程称其为“**容器化技术**”，实际上就是将支持某个项目（比如 MC 服务端）运行的最小范围的内容，打包成一个所谓**镜像文件**（image），并且可以上传到网络平台上，利用平台为镜像文件提供托管，供其他人方便部署、复用。别人部署的过程，就是所谓的**拉取镜像**；别人复用的过程，就是所谓的**运行容器**（**运行实例**）的过程。也就是说，Docker 镜像文件类似于集装箱，而 Docker 镜像文件内部的项目，就是放在其中的货物。集装箱（Docker 镜像文件）提供标准化的托管，并且不同集装箱之间互不影响。

譬如，对镜像提供者来说，在自己 PC 上构建一个应用程序，将其依赖关系一起打包到一个 Docker 镜像文件中，这镜像就很容易通过托管平台运送到别人的 PC 或者服务器上运行，易于装载、复制、移除。对镜像拉取者来说，从托管镜像文件的平台上拉取镜像以后，通过图形界面或者指令集运行这个镜像文件，就会以这个镜像文件为**模板**创建一个**容器**（也叫**实例**），并开启一个进程去运行这个实例提供的服务。从这里也能看出常说的**镜像**跟**容器**的区别：镜像是一个**模板**，只有可读层，**只可读不可写**；容器是一个根据模板生成的**实例**，在镜像的可读层上加了一层可写层，**可读可写**。

**举个例子：**
MC 服务端的运行需要 Java 支持。如果想开一个小服务器玩，我们首先检查 MCSM 面版里有没有 MC 某版本对应的 Java 文件（譬如，Minecraft 1.20.1 对应 Java 17）。如果没有，我们就需要从 Java 的提供方—— Oracle 官网拉取内含 Java 文件的镜像：

```bash
docker pull container-registry.oracle.com/graalvm/jdk:17
```

通过该命令，我们从 Oracle 官网拉取了上面存储的镜像文件并下载到本地，其中包含编译、部署Java应用所需的所有内容。

相比虚拟机，Docker 镜像文件不需要内置一个完整的操作系统，显得更轻量化，而且有方便的图形界面、指令集进行管理，并且与 MCSM 管理面板集成度更高。对比来说，Docker 和虚拟机都是对某服务的**封装**技术。借助这些封装技术，我们不需要详细了解服务提供者的系统配置和环境需求，就可以运行提供的服务。这为团队协作和实际部署提供了很大的便利，同时也方便了服务提供者自身对于项目的管理。**封装**这一概念早已深入现代的计算机技术的方方面面，小到语法层面具体函数和类的封装，到项目层面各类库的封装，到应用层面对各种服务（比如虚拟机和 Docker 镜像）的封装，还有计算机系统内部对于物理硬件、输入输出架构、操作系统的封装，以及计算机网络中对数据链路、通信协议的封装，使得开发者不需要详细了解底层是如何实现的，只需要积极搜索教程或询问 AI，找到底层提供服务的入口——**接口**（API）（比如 Docker 提供的图形界面或者指令集就是接口），就可以在简单运用 API 的基础上进行开发和应用。这种方法论为不同层面的开发者提供了极大便利。这也是我们作为初入此领域的萌新所需重点关注的理念，这会对日后的学习有极大的帮助。
